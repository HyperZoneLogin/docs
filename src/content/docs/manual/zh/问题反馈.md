# 问题反馈

当你在使用 HyperZoneLogin 过程中遇到问题，建议先做一次基础自检，再选择合适的反馈渠道。这样既能提高问题被定位的效率，也能避免把本来属于环境配置的问题误判成插件 Bug。

## 反馈前先自检

在提交问题之前，建议先确认以下内容：

- [ ] 你已经阅读过本手册对应章节；
- [ ] 你已经确认 Velocity 与后端服基础转发配置无误；
- [ ] 你知道自己安装了哪些模块；
- [ ] 你能提供复现步骤，而不只是“不能用”；
- [ ] 你已经查看过启动日志或报错信息；
- [ ] 如果问题与迁移相关，你已经备份过原始数据。

## 建议附带的信息

无论你是在 GitHub 还是 QQ 群反馈，尽量一次性提供下面这些信息：

1. **使用的版本**
   - `HyperZoneLogin` 主插件版本
   - 所有已安装子模块版本
   - Velocity 版本
   - 后端服务端版本

2. **部署结构**
   - 是否使用 Velocity
   - 是否使用 `modern forwarding`
   - 是否安装了 `auth-yggd`、`auth-offline`、`data-merge`、`profile-skin`

3. **问题现象**
   - 玩家在哪一步出问题
   - 是首次登录失败、迁移失败、皮肤异常，还是跳服异常
   - 是否能稳定复现

4. **日志与配置**
   - 相关报错日志
   - 涉及的配置片段
   - 是否改动过默认配置

5. **复现步骤**
   - 玩家如何进入
   - 执行了什么命令
   - 预期结果是什么
   - 实际结果是什么

这些信息越完整，问题越容易被准确判断。

## GitHub 反馈

根据根目录 `README.md`，项目接受通过 GitHub 提交问题。对使用者来说，最适合在以下情况中使用 GitHub：

- 能稳定复现的问题；
- 有明确日志、配置与步骤的 Bug；
- 与某个模块行为明显相关的问题；
- 文档缺失或描述不清的问题。

### 建议的 GitHub 提问方式

- 标题写清楚模块与现象；
- 正文按“环境 → 步骤 → 现象 → 日志 → 配置片段”的顺序描述；
- 不要只写一句“用不了”；
- 不要省略版本信息。

## 哪些问题更容易得到帮助

通常更容易被处理的问题包括：

- 有清晰日志的报错；
- 有明确复现步骤的异常行为；
- 迁移配置填写后无法执行；
- 某个已知模块在特定条件下行为异常；
- 文档与实际配置不一致。

## 哪些问题需要你先补充信息

以下情况通常需要你先整理环境信息，再继续反馈：

- 只说“进不去服”；
- 只说“插件没反应”；
- 完全不提供版本、日志、配置；
- 没有说明自己到底安装了哪些模块；
- 连基础代理转发是否配置正确都还没确认。

这不是为了提高门槛，而是因为登录框架的问题往往横跨代理、后端、数据库与认证服务。没有上下文，别人几乎无法准确判断。

## 支持边界的正确理解

本项目现有文档与提示词已经表达了一个很明确的边界：

- 项目会接受问题反馈；
- 但前提是你已经完成基本的自查；
- 对于缺乏基础上下文的问题，通常很难直接给出有效结论；
- 文档已覆盖的内容，优先以文档为准；
- 文档未明确写出的部分，不应默认一定存在额外的口头支持。

换句话说，**先阅读、先自检、再反馈**，会让你更快得到真正有用的帮助。

## 一个推荐模板

你可以直接按下面的格式整理问题：

```text
环境：Velocity 版本 / 后端版本 / HyperZoneLogin 版本 / 已安装模块
现象：发生了什么问题
步骤：如何稳定复现
日志：关键报错内容
配置：相关配置片段
已尝试：你已经做过哪些排查
```

如果你能把问题整理到这个程度，无论是发 Issue 还是去 QQ 群交流，效率都会高很多。
