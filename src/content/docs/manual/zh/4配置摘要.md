# 4. 配置摘要

这一章按照 `用户手册/ai生成词/config/` 中现有配置样例，逐个说明各文件的作用与修改时机。

先给一个总原则：**默认配置通常是为了“先跑起来”而准备的。** 如果你已经能正常完成登录流程，很多配置都不需要急着修改；但如果你决定改动，请先确认你理解该项会影响哪一段认证链路。

## 4.1 `database.conf`

作用：定义主插件使用的数据库类型、连接方式、表前缀与连接池参数。

### 已确认的关键项

- `type`：支持 `SQLITE`、`MYSQL`、`MARIADB`
- `sqlite.path`：SQLite 数据文件路径
- `mysql.*` / `mariadb.*`：主机、端口、库名、用户名、密码与 JDBC 参数
- `table-prefix`：所有表的统一前缀
- `pool.*`：连接池大小、空闲数、超时与生命周期

### 使用建议

- **初次部署**：可以先按默认 `SQLITE` 跑通流程；
- **正式环境**：如果你已经有稳定数据库服务，再考虑切换到 MySQL / MariaDB；
- **旧 H2 数据迁移**：如果你需要读取历史 H2 数据，请使用 `data-merge` 模块，而不是把核心数据库类型改成 H2。

### 修改风险

- 更换数据库类型前，应先做好数据备份；
- 修改 `table-prefix` 会影响所有表名，不适合在已有正式数据时随意变更；
- 数据库连接信息填错会导致主插件无法正常初始化。

## 4.2 `misc.conf`

作用：控制若干全局杂项行为。

### 已确认的关键项

- `debug=true`：是否开启调试模式
- `enable-replace-game-profile=true`：是否启用 `GameProfile` 替换
- `kill-chat-session=true`：是否不向服务器发送 `CHAT_SESSION_UPDATE` 包

### 使用建议

- 初次排障阶段保留默认值更方便定位问题；
- 如果你的环境已经稳定，是否关闭 `debug` 可以再结合实际日志量决定；
- 其余两项直接影响代理到后端的档案 / 会话行为，不建议在不了解后果时随意改动。

### 修改风险

`misc.conf` 中的开关往往不是“界面风格选项”，而是认证链路的底层行为开关。改动之前，务必先确认你知道它影响的是哪一段流程。

## 4.3 `backend-server.conf`

作用：配置未安装 `limboapi` 时，真实后端认证等待区与认证后默认跳转逻辑。

### 已确认的关键项

- `fallback-auth-server=lobby`
- `post-auth-default-server=play`
- `remember-requested-server-during-auth=true`

### 使用建议

- 如果你没有 `limboapi`，请明确指定一个真实后端服作为认证等待区；
- `post-auth-default-server` 适合填写认证成功后最常进入的目标服；
- 如果你希望保留玩家认证期间临时选择的目标服，可保留 `remember-requested-server-during-auth=true`。

### 修改风险

- 填写了不存在的服务器名，会导致认证前后跳转逻辑不符合预期；
- 这几个值直接依赖你的 Velocity 服务器列表命名，必须与代理实际配置保持一致。

## 4.4 `remap.conf`

作用：控制 UUID 生成时使用的前缀。

### 已确认的关键项

- `prefix=HyperZone`

### 使用建议

如果你没有非常明确的兼容需求，建议保持默认值。

### 修改风险

该项会影响 UUID 生成规则。对于已经运行并存在历史数据的环境，中途修改可能造成映射不一致，因此不建议在生产环境里随意调整。

## 4.5 `offlinematch.conf`

作用：控制离线识别与离线匹配策略。

### 已确认的关键项

- `enable=true`
- `uuid-match.zero=true`
- `uuid-match.offline=true`
- `uuid-match.pcl2.enable=true`
- `uuid-match.pcl2.hash=true`
- `uuid-match.pcl2.slim=true`
- `host-match.start=[offline, o-]`

### 使用建议

这个文件主要与离线识别逻辑相关。只有在你确实需要处理离线玩家、特定启动器特征或特定域名前缀识别时，才需要深入调整。

### 修改风险

- 过度放宽匹配规则，可能导致错误识别；
- 贸然缩紧匹配规则，也可能让原本可识别的离线接入方式失效；
- `host-match` 属于接入入口识别的一部分，修改前应先梳理你的域名与入口规划。

## 4.6 `entry/mojang.conf`

作用：定义 Mojang 官方 Yggdrasil 验证入口。

### 已确认的关键项

- `id=mojang`
- `name="Mojang Official"`
- `yggdrasil.url` 指向官方 `hasJoined` 校验地址
- `timeout`、`retry`、`retry-delay`
- `proxy.type`、`hostname`、`port`、`username`、`password`

### 使用建议

- 如果你需要官方正版验证，这个配置通常是基础入口；
- 默认情况下可先直接使用；
- 如果你的网络环境需要代理，再根据实际情况调整 `proxy` 段。

### 修改风险

- `url` 模板中的 `{username}`、`{serverId}`、`{ip}` 占位符不能乱改；
- 代理设置填写错误会导致请求超时或验证失败。

## 4.7 `entry/littleskin.conf`

作用：定义 Little Skin 的 Yggdrasil 验证入口。

### 已确认的关键项

- `id=littleskin`
- `name="Little Skin"`
- `yggdrasil.url` 指向 Little Skin 的 `hasJoined` 接口
- 同样支持超时、重试与代理配置

### 使用建议

如果你要接入 Little Skin，可直接参考该配置；如果你没有相关需求，可以保留文件但不必额外扩展。

## 4.8 `entry/example/example.conf`

作用：提供自定义 Entry 模板。

### 重要说明

样例文件中已经明确写出：**位于 `example/` 子目录中的配置不会被加载。**

正确做法是：

1. 复制一份模板到 `entry/` 根目录；
2. 修改 `id`、`name`、`url` 等参数；
3. 重载或重启后再让模块读取。

## 4.9 `merge/merge-ml.conf`

作用：配置从 MultiLogin / MinecraftLink 旧数据迁移到 HyperZoneLogin 的方式。

### 已确认的关键项

- `source.type`：支持 `H2DB` 或 `MYSQL`
- `source.h2.*`：H2 路径、参数、用户名、密码
- `source.mysql.*`：MySQL 连接信息
- `tables.user-data-table`
- `tables.in-game-profile-table`
- `service-id-mapping`

### 使用建议

- 仅在安装了 `data-merge` 且确实需要迁移 MultiLogin 数据时使用；
- 执行前先确认旧库表名与配置一致；
- `service-id-mapping` 用于把旧系统的 `service_id` 映射到新的 `entryId`，这是迁移成功与否的关键之一。

### 修改风险

- 表名写错会直接导致迁移读取不到数据；
- `service-id-mapping` 配错可能让 Entry 归属错误；
- 迁移前不备份，出了问题很难回退。

## 4.10 `merge/merge-am.conf`

作用：配置从 AuthMe 迁移离线账号数据。

### 已确认的关键项

- `source.type`：支持 `SQLITE` 或 `MYSQL`
- `source.sqlite.*`：SQLite 路径与参数
- `source.mysql.*`：MySQL 连接信息
- `tables.auth-me-table`

### 使用建议

- 仅在你确实要接管 AuthMe 历史账号时使用；
- 迁移前先确认旧库类型与表名；
- 迁移完成后，要核对密码格式兼容情况与迁移日志。

### 修改风险

- 数据库连接填错会导致迁移命令无法执行；
- 表名错误会导致找不到 AuthMe 数据；
- 未核对密码格式就直接上线，会给玩家登录造成风险。

## 4.11 `profile-skin.conf`

作用：控制 `profile-skin` 模块的皮肤提取、修复、缓存与应用行为。

### 已确认的关键项

- `enabled=true`
- `prefer-upstream-signed-textures=true`
- `trusted-signed-texture-entries=[mojang]`
- `restore-unsigned-textures=true`
- `allow-initial-profile-fallback=true`
- `mine-skin.method=URL`
- `mine-skin.url-endpoint`
- `mine-skin.upload-endpoint`
- `mine-skin.timeout-millis`
- `mine-skin.user-agent`

### 使用建议

- 如果你安装了 `profile-skin`，建议先保持默认值，观察皮肤链路是否稳定；
- `trusted-signed-texture-entries` 适合填写你明确可信任的上游来源；
- 只有在你清楚 MineSkin 修复流程时，才建议进一步调整接口与修复策略。

### 修改风险

- 过度信任不可靠的上游签名源，可能导致皮肤链路质量不稳定；
- 关闭修复或回退逻辑后，某些玩家可能失去原本可用的皮肤数据；
- 更改 MineSkin 接口时，要确认网络可达性与接口行为一致。

## 4.12 一条实用原则：先生成、再理解、后微调

对于 HyperZoneLogin 这类模块化登录框架，最稳妥的配置方式通常是：

1. 先按默认值生成全部配置；
2. 只改与你当前部署模型直接相关的那几项；
3. 每次只改一类配置，并做验证；
4. 对涉及身份映射、UUID 规则、迁移与皮肤来源的配置保持谨慎。

如果你已经能正常完成登录、识别与转发，默认配置通常就足够作为第一阶段的可用方案。
